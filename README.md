# ContextAgent

ContextAgent 是一个基于 openJiuwen 框架构建的上下文代理，目标是为业务 Agent 或大模型调用链提供统一、可扩展、可治理的上下文能力，帮助系统更准确地理解用户意图、任务背景与历史状态。

项目参考了 Claude Code、OpenClaw 等 Agent 系统中的上下文组织思路，重点解决以下问题：

- 如何将长期记忆、会话历史、用户偏好和外部知识统一组织为可消费的上下文。
- 如何同时兼容 openJiuwen 内置记忆系统与外部记忆基础设施，避免能力绑定在单一存储实现上。
- 如何在多轮交互过程中动态更新上下文，保持信息相关性并控制上下文长度。
- 如何为其他 Agent 提供稳定的上下文读取、压缩、摘要和注入能力。
- 如何让上下文压缩、选择、裁剪等算法能够按业务场景灵活定制，而不是依赖单一固定策略。
- 如何遵循上下文工程的核心原则，只向模型提供最小但足够高信号的上下文集合，避免上下文污染与注意力浪费。
- 如何在预取式召回之外，支持 just-in-time 的按需上下文检索，让 Agent 通过工具自主探索信息空间。
- 如何为长时任务提供压缩、结构化笔记、子代理协作等机制，维持跨阶段一致性。
- 如何系统性处理上下文失效问题，例如 context poisoning、context distraction、context confusion 和 context clash。
- 如何在保证召回质量的同时，将上下文召回时延稳定控制在低水平，满足在线调用场景。

## 项目目标

- 构建一个独立的 Context Agent，作为多 Agent 系统中的上下文中枢。
- 为上层 Agent 提供标准化的上下文组装、检索、压缩与注入接口。
- 兼容 openJiuwen 记忆系统，并支持对接外部向量数据库、图数据库等记忆系统。
- 建设分层分级的记忆体系，在不同成本、时效和精度之间做合理分工。
- 支持按照业务场景定制上下文压缩、选择和组装策略。
- 建立面向 Agent 的上下文工程能力，确保上下文是被持续治理和按需加载的，而不是静态堆叠的。
- 在效果、性能与可维护性之间建立平衡，避免上下文无限膨胀或信息失真。
- 将关键路径上的上下文召回时延作为核心指标，目标控制在 300ms 以内。

## 核心能力规划

围绕业界常见的上下文工程方法，ContextAgent 的能力可归纳为四类：

- Write：把关键上下文写出到主上下文窗口之外
- Select：在需要时把相关上下文选择性拉回窗口内
- Compress：对已有上下文进行压缩、裁剪和提炼
- Isolate：通过子代理、子任务或边界隔离分散上下文负载

### 1. 多源上下文聚合

支持整合多类上下文信息，包括但不限于：

- 对话历史
- 用户偏好
- 长期记忆
- 任务状态
- 外部知识与工具返回结果

在数据源层面，系统既可复用 openJiuwen 提供的记忆能力，也可按需接入外部存储与检索系统，例如：

- 向量数据库
- 图数据库
- 关系型或文档型存储
- 其他业务侧已有记忆服务

目标是通过统一抽象屏蔽底层差异，让调用方以一致方式访问不同来源的上下文信息。

### 2. 分层分级记忆体系

围绕不同类型上下文的时效性、重要性与访问成本，建设分层分级的记忆系统。例如：

- 热层：当前会话、近期任务状态、即时用户偏好，强调低时延与高命中。
- 温层：阶段性摘要、近期长期记忆，强调可回溯与性价比。
- 冷层：深历史知识、低频关系数据、外部知识库，强调容量与可扩展性。

系统应支持根据任务类型、检索预算和响应时延要求，动态决定召回路径与记忆层级。

同时，记忆内容在语义上也可进一步区分为：

- Procedural memory：规则、指令、偏好、约束等“怎么做”的信息
- Episodic memory：历史案例、few-shot 示例、过往任务片段等“曾发生过什么”的信息
- Semantic memory：事实、实体、关系、领域知识等“知道什么”的信息

系统应允许不同类型记忆采用不同的存储、索引、选择和注入方式，而不将其混为单一数据池。

### 3. 动态上下文管理

根据对话阶段、任务目标和调用方需求，动态调整上下文内容，确保进入模型的信息尽可能相关、精炼、可解释。

在实现上，应明确将上下文视为有限且会衰减的注意力预算，而不是可以无限堆叠的缓存空间。系统需要持续评估哪些信息应保留、哪些应压缩、哪些应延迟到运行时再检索。

### 4. 即时上下文检索与 Agentic Search

除传统的预计算检索外，系统应支持 just-in-time 的上下文检索模式。即调用方或 Agent 不必在推理前一次性加载所有候选信息，而是先携带轻量引用，再通过工具在运行时逐步拉取高价值上下文，例如：

- 文件路径、对象 ID、查询模板、知识条目引用
- 向量召回后的候选引用
- 图关系查询后的节点或边引用
- 外部系统中的文档链接与结果指针

该能力应支持 Agent 基于任务推进逐步探索信息空间，减少无效上下文注入，提升对大型知识库、长任务链路和复杂工程环境的适应性。

### 5. 混合式召回策略

系统不应局限于单一召回路径，而应支持混合策略：

- 对稳定、高频、低成本的信息进行前置注入
- 对体量大、变化快、命中不确定的信息采用按需检索
- 根据时延预算、任务复杂度和历史命中效果，在预取与运行时探索之间动态切换

这种混合模式更适合真实 Agent 场景，能够在低时延要求与复杂任务理解之间取得平衡。

### 6. 选择策略与上下文暴露控制

系统应支持对不同来源上下文的精细暴露控制。例如：

- 控制哪些 scratchpad 字段在当前步骤可见
- 控制哪些记忆类别允许进入当前模型调用
- 控制哪些工具描述在当前任务中暴露给 Agent
- 控制哪些中间结果仅保存在状态层，而不直接注入模型上下文

这一能力的核心目的是减少“上下文窗口不再属于当前任务”的问题，避免无关信息意外干扰当前推理。

### 7. 面向 Agent 的上下文接口

为其他 Agent 提供可调用接口，使其能够按需获取上下文快照、摘要结果、记忆检索结果或压缩后的任务背景。

### 8. 记忆处理与异步更新

支持对长期记忆等数据源进行异步处理，并将处理结果同步到 Agent 或模型可访问的消息上下文中。

### 9. 上下文压缩与摘要

对历史对话、任务轨迹和冗余信息进行压缩、提炼与总结，提升模型对长程信息的利用效率。

在策略层面，应支持按场景定制不同的上下文处理算法，例如：

- 面向问答场景的高相关片段选择
- 面向任务执行场景的状态保留与过程裁剪
- 面向长会话场景的滚动摘要与阶段性压缩
- 面向高实时场景的低成本快速筛选策略

系统不应将压缩、选择、裁剪逻辑固化为单一实现，而应提供可插拔、可配置的策略机制，支持不同业务根据效果、成本和时延目标进行选择。

对于长时任务，应重点支持高保真 compaction 机制：在接近上下文上限时对历史内容进行压缩，同时尽可能保留架构决策、未解决问题、关键约束和当前执行状态，并清理重复消息和低价值工具结果。

除全局压缩外，系统还应支持边界压缩与局部压缩，例如：

- 在高 token 工具调用后对结果进行局部摘要
- 在 agent-agent handoff 时生成紧凑交接摘要
- 在阶段切换、子任务完成、长链路执行中生成阶段性摘要
- 在超过预算前执行启发式 trimming 或 pruning

### 10. 结构化笔记与外部工作记忆

除压缩外，系统还应支持结构化笔记机制，将关键中间状态持久化到上下文窗口之外，例如任务计划、阶段性结论、关键依赖、未决事项、风险项等，并在后续阶段按需重新注入。

这类“工作记忆”应与长期记忆、会话记忆区分管理，既保证低开销持久化，也避免把所有过程细节长期保留在主上下文中。

scratchpad 既可以实现为文件型记忆工具，也可以实现为运行时状态对象中的结构化字段。系统应同时支持这两类实现，以适配不同部署形态。

### 11. 工具上下文治理

工具本身也是上下文系统的一部分。系统应支持：

- 只向当前任务暴露必要工具集合
- 对重叠功能工具做去歧义与路由控制
- 对工具描述、参数说明和返回结构进行标准化
- 对大规模工具集使用检索式工具选择，而不是一次性注入全部工具说明

在复杂场景下，可以对工具描述做 RAG 检索或路由筛选，减少工具选择歧义，提升调用准确率。

### 12. 结构化存储与快速访问

设计清晰的上下文数据结构和存储机制，兼顾查询效率、扩展性与后续治理能力。

对于大规模知识库、代码库或复杂业务数据，检索链路可组合使用：

- embedding 检索
- grep 或关键字检索
- 文件系统导航与层级信号
- 图关系检索
- rerank 重排序

避免单纯依赖 embedding 作为唯一召回手段。

### 13. 版本管理与可回滚

支持上下文版本化管理，在关键节点保留快照，便于调试、回放与回滚。

### 14. 子代理与上下文隔离

针对复杂任务，可引入子代理架构。主代理负责规划、协调和结果汇总，子代理在相对独立的上下文窗口中执行检索、分析、总结或专项推理，并仅返回压缩后的结果。

这种方式可以避免单一上下文窗口承载全部探索细节，降低上下文污染风险，同时提升复杂分析、并行探索和长链路任务的可扩展性。

### 15. 多语言与多模态扩展

预留对多语言文本及多模态上下文的处理能力，为复杂业务场景提供扩展基础。

## 典型应用场景

- 为任务执行型 Agent 提供完整的历史背景与用户偏好信息。
- 为问答型 Agent 注入外部知识、任务状态与摘要后的上下文。
- 为编排型 Agent 提供跨步骤共享的统一上下文视图。
- 为需要长期记忆能力的系统提供稳定的上下文更新与读取机制。
- 为长时执行任务提供压缩续跑、结构化笔记和子代理协同能力。
- 为大型代码库、知识库或复杂业务系统提供按需探索式上下文检索能力。

## 设计原则

- 最小高信号优先：进入模型的上下文应尽可能少，但必须足够支撑当前任务。
- 相关性优先：仅注入与当前任务强相关的信息。
- 上下文所有权优先：应尽量确保上下文窗口服务于当前任务目标，避免历史记忆或无关召回意外劫持当前推理。
- 可解释性优先：上下文来源、结构与变更应尽量可追踪。
- 可扩展性优先：能够平滑接入新的上下文类型与数据源。
- 分层治理优先：根据记忆价值、热度和访问成本对信息进行分层分级管理。
- 策略可定制优先：压缩、选择、裁剪和注入策略应支持按场景配置与替换。
- 工具高效优先：上下文工具与检索接口应返回足够解决问题、但不过量的结果，避免工具输出本身成为噪声来源。
- 渐进披露优先：优先支持逐步发现和逐步注入上下文，而非一次性加载所有候选信息。
- 失败模式可控优先：应重点防范上下文污染、干扰、混淆和冲突四类典型失效模式。
- 性能优先：在上下文质量与系统响应速度之间保持合理平衡。

## 性能目标

- 上下文召回链路以低时延为核心约束，目标是关键路径召回耗时不超过 300ms。
- 在架构设计上优先采用缓存、分层检索、异步预处理和增量更新等手段，避免在请求路径上执行高成本全量扫描。
- 对外部向量数据库、图数据库等依赖应保留降级策略，确保单点波动不会显著放大整体时延。
- 对工具调用与运行时检索返回结果施加 token 预算与裁剪策略，避免工具结果膨胀导致主上下文退化。

## 上下文工程要求

结合 Anthropic、Claude Code 与业界实践，ContextAgent 在需求层面应满足以下上下文工程要求：

1. 将上下文视为有限的注意力预算，而不是无限缓存空间。
2. 以“最小但足够”的高信号上下文组装为默认原则，严格控制噪声注入。
3. 支持系统提示、工具定义、示例、消息历史、外部检索结果等不同上下文组件的统一治理。
4. 支持 canonical few-shot 示例管理，优先保留高代表性的典型样例，而非堆叠大量边缘案例。
5. 支持 just-in-time 检索与 agentic search，让 Agent 在运行时自主拉取所需上下文。
6. 支持混合式上下文策略，在前置召回与运行时探索之间按场景动态平衡。
7. 支持 procedural、episodic、semantic 等不同记忆类型的差异化管理。
8. 支持 scratchpad、会话状态、外部记忆文件等多形态工作记忆载体。
9. 支持对长任务进行高保真 compaction，优先保留关键决策、约束、未决问题和当前进度。
10. 支持局部压缩、边界压缩、启发式 trimming 与工具结果 pruning 的组合策略。
11. 支持结构化笔记与外部工作记忆，以低成本方式跨阶段保存关键状态。
12. 支持子代理上下文隔离与摘要回传，避免复杂任务的探索细节污染主上下文。
13. 支持检索式工具选择与工具集裁剪，降低工具重叠和路由歧义。
14. 支持 embedding、关键词检索、层级信号、图关系与 rerank 的组合召回。
15. 支持对工具输出进行清理、压缩和预算控制，避免原始结果长期滞留在主上下文窗口中。
16. 支持基于评测持续优化上下文策略，关注召回质量、压缩质量、时延、成本和任务成功率。

## 典型风险

- Context poisoning：错误信息进入上下文后被持续继承和放大。
- Context distraction：上下文过多导致模型注意力分散。
- Context confusion：无关或冗余信息干扰当前任务判断。
- Context clash：多个上下文片段相互冲突，导致模型行为不稳定。

ContextAgent 的设计应默认面向这些风险做检测、隔离、压缩和治理。

## 预期价值
